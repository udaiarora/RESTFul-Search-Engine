APPENDIX II
VAX Transition Appendix


V.1 Inventory of VAX Files

Your VAX files can exist on either disk or tape. To list all the files and their sizes in your disk space, including those in subdirectories, type:

dir/siz [...]

To list a summary of the number of files on your account, and the space they occupy, type:

dir [...] /siz/tot

This will produce a listing something like:

Directory DISK1:[ACAD.AGJES]
Total of 4 files, 62 blocks.

Directory DISK1:[ACAD.AGJES.MAIL]
Total of 424 files, 4065 blocks.

Directory DISK1:[ACAD.AGJES.TEST]
Total of 10 files, 1656 blocks.

Grand total of 4 directories, 454 files, 5783 blocks.

VAX file sizes are in blocks of 512 bytes (or 1/2 a Kilobyte). To calculate a "ballpark" figure on the amount of space they occupy, divide the number of blocks by two, to get the size in Kilobytes. Then divide that by 1000 to get the size in Megabytes.

Using the example above:

5783 blocks / 2 = 2891.5 KB ;
2891.5 / 1000 = 2.89 MB

Thus, you'd need about 3 MB of disk space in AFS to move all these files.

On ACVAX there is no automatic way of knowing whether you have files on tape. However, you can browse the file ASU$SUPPORT:TAPES.DAT to see whether you own any tapes. Note that all VAX tape files must be moved, i.e., read to disk and FTPed, before the VAX is removed. Most VAX tapes in the ASU tape library are in VAX VMS backup format and can only be read by a VAX computer. After ACVAX is removed, you may have no other recourse than to use a service bureau to transfer your tape files. For an example on using the BACKUP command, see the file ASU$SUPPORT:BACKUP.EX .

V.2 File Types

Many VAX files can be categorized into one of eight types: source code files (.FOR, .C, .PAS), object files (.OBJ), executables (.EXE), script files (.COM), data files (.DAT), directories (.DIR), mail files (.MAI), and other text files (.LIS, etc.). Executables, object files, script files, directory files and mail files are VAX-specific and should not be moved to a non-VAX platform. Source code files, ASCII data files, and other text files (.LIS, etc.) can be moved by FTPing or downloading to a new platform.

V.3 Legacy System Software

Most VAX software applications have functional replacements on alternate platforms. See Appendix A for a list of these.

V.5 Porting Programs You've Written

The binary executable files that you've created from source code programs compiled and linked on ACVAX will not work on other non-VAX platforms. You will have to move your source code files to a new platform and re-compile/re-link them. You may be required to change some statements in your existing code to be compatible under the new target platform's compiler. For example, using "SHARED" mode in a file OPEN statement on ACVAX may not be allowed on some UNIX systems.

Equivalent compilers for those on the VAX are available on alternate platforms:

V.6 Moving Files

The simplest and easiest way to move files between platforms is to use FTP (File Transfer Protocol). An example session follows. Note that the default transfer mode is ASCII. To switch to binary mode, type: bin. Also, to transfer multiple files, use "mput" rather than "put", e.g., "mput *.dat".

AGJES> ftp research.asu.edu
ACVAX.INRE.ASU.EDU MultiNet FTP user process 3.3(109)
Connection opened (Assuming 8-bit connections)
research1.asu.edu FTP server ready.
FTP>user agjes
Password required for agjes.
Password:
User agjes logged in.
FTP>put test.dat test.dat
Opening ASCII mode data connection for test.dat.
Transfer complete.
FTP>quit
Goodbye.
AGJES>

V.7 Mail Files

Moving mail files is both difficult and labor-intensive, and we recommend you print the important ones and leave the files behind. If there are mail messages that you must save in electronic form, you can do so with the VAX Mail "EXTRACT" command. It will save the mail note as an external ASCII file on your VAX disk space, which you can then FTP. For example, after reading a mail note, type: EXTRACT. It will prompt for a file name and then save the current note under that file name:

MAIL>extract
_File: mailmsg1.not
%I-CREATED, DISK1:[ACAD.AGJES]MAILMSG1>NOT;1 created

Additional information on the EXTRACT command is available by typing HELP EXTRACT while in Mail.
